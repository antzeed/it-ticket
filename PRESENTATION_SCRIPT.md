# 🎤 สคริปต์และคู่มือสำหรับการนำเสนอโปรเจค (Presentation Script & Q&A)

เอกสารฉบับนี้ถูกเตรียมไว้สำหรับการซ้อมนำเสนอ (พรีเซนต์) โปรเจคระบบ IT Helpdesk โดยแบ่งเป็น "สไลด์/หน้าตามหัวข้อ" พร้อม "บทพูด" (Script) และ "คำถามที่อาจารย์มักจะดักถาม พร้อมคำตอบ" (Q&A Cheat Sheet) เพื่อช่วยให้ผู้พูดมีความมั่นใจสูงสุด ไม่ว่าอาจารย์จะถามเจาะลึกแค่ไหนก็ตาม

---

## 📍 ส่วนที่ 1: แนะนำโปรเจค (Introduction)

**🗣️ สิ่งที่ต้องพูดพรีเซนต์:**
> "สวัสดีครับ/ค่ะ วันนี้จะขอนำเสนอโปรเจคระบบ IT Helpdesk ซึ่งเป็นแพลตฟอร์มสำหรับจัดการการแจ้งซ่อมและปัญหาไอทีภายในองค์กรครับ 
> 
> เป้าหมายหลักของเราคือการเปลี่ยนรูปแบบการแจ้งซ่อมแบบเก่า (เช่น ผ่านไลน์กรุ๊ป หรือเขียนกระดาษ) ซึ่งมักจะตกหล่นหรือตามเรื่องยาก มาเป็นระบบที่ **รวมศูนย์ (Centralized)** ติดตามสถานะได้อย่างชัดเจนตั้งแต่วันเปิดตั๋ว (Open) จนถึงปิดงาน (Closed) รวมถึงมีการจัดเก็บข้อมูลอย่างเป็นระบบเพื่อให้ฝ่าย IT นำไปวิเคราะห์ต่อยอดได้ครับ"

## 📍 ส่วนที่ 1.1: สถาปัตยกรรมระบบ (Architecture) และคลาวด์ (Cloud Infrastructure)

**🗣️ สิ่งที่ต้องพูดพรีเซนต์:**
> "ในด้านโครงสร้างของโปรเจคนี้ เราแบ่งการอธิบายออกเป็น 2 มุมมองหลักครับ คือฝั่ง Code และ ฝั่ง Cloud:
>
> **1. สถาปัตยกรรมฝั่งโค้ด (Code Architecture):**
> เราออกแบบระบบในลักษณะ **Client-Server Architecture** ครับ โดยแยกผู้ขอบริการ (Frontend) และผู้ให้บริการประมวลผล (Backend) ออกจากกันอย่างเด็ดขาด ช่วยให้ในอนาคตถ้าเราอยากทำ Mobile App ก็สามารถมาต่อช่องทาง Backend เดิมได้เลยโดยไม่ต้องแก้โค้ดใหม่ทั้งหมด 
>
> **2. สถาปัตยกรรมฝั่งคลาวด์ (AWS Cloud Architecture):**
> การนำระบบขึ้นออนไลน์ เราใช้เทคโนโลยี **Docker** รวบรวมโค้ดให้กลายเป็นอิมเมจ (Image) แล้วนำไปวางบนคลาวด์ของ **AWS (Amazon Web Services)** เต็มรูปแบบ:
> - คอนเทนเนอร์ (API/Web) จะถูกดึงไปรันบน **ECS (Fargate)** ที่ดูแลเซิร์ฟเวอร์สเกลอัตโนมัติแบบ Serverless
> - ข้อมูลรูปภาพการแจ้งซ่อมถูกฝากส่งไปเข้า **Amazon S3** แยกเอาไว้เพื่อไม่ให้ Database บวม
> - ความพิเศษคือ แทนที่เราจะเช่า Load Balancer (ALB) หรือ Route53 ของระบบ AWS ที่ราคาแพง เราเปลี่ยนมาเชื่อมต่อระบบของเราออกสู่อินเทอร์เน็ตผ่าน **Cloudflare Tunnel** ครับ ซึ่งให้ความปลอดภัยสูงมาก ปิดซ่อน IP จริงของเซิร์ฟเวอร์เราทั้งหมด แถมยังให้บริการจัดการ Load Balance และโดเมนให้แบบยืดหยุ่นและปลอดภัยกว่าเดิมครับ!"

**💡 คำถามดักทางที่อาจารย์มักจะถาม (Q&A):**
- **Q: ทำไมเลือกใช้ Cloudflare Tunnel ทำไมไม่เปิดพอร์ตให้เข้าจากเน็ตตรงๆ ให้ ALB / Route53 จัดการ?**
  - **A ตอบอาจารย์:** "การใช้ Cloudflare Tunnel เป็นแนวทาง Zero Trust Security ครับ คอมพิวเตอร์หรือคอนเทนเนอร์ฝั่งเราจะทำการเจาะท่อ (Tunnel) ทะลุออกไปหา Cloudflare จากด้านในเอง (Outbound) ทำให้เราไม่ต้องตั้งค่าเปิดไฟร์วอลรับคนจากข้างนอก (Inbound Ports) เพื่อรับการโจมตีเลย แถมยังได้ฟีเจอร์กันโดนถล่มเว็บ (DDoS Protection) ของ Cloudflare แบบเต็มพิกัด รวมถึงถูกกว่าการไปเช่าเซิร์ฟเวอร์ ALB ของ AWS ด้วยครับ!"
- **Q: Docker คืออะไร ไม่ใช้ไม่ได้หรอ ก็โยนโค้ดขึ้นเซิร์ฟเวอร์เหมือนเมื่อก่อน?**
  - **A ตอบอาจารย์:** "ปัญหาคลาสสิคคือ 'รันเครื่องผมได้ แต่รันเครื่องเซิร์ฟพัง' Docker แก้ปัญหานี้ด้วยการ 'แพ็คทุกอย่างใส่กล่อง' ครับ ไม่ว่า OS, โค้ด, หรือเซ็ตอัพ กลายเป็นว่าไม่ว่าจะยกไปรันที่ไหนระบบก็จะได้ผลลัพธ์เหมือนกันเป๊ะ รันได้เสมอครับ"
- **Q: ทำไมต้องเอาไฟล์รูปภาพโยนไปเก็บที่ S3 เปลืองไหม ทำไมไม่เก็บลง Database?**
  - **A ตอบอาจารย์:** "ถ้ายัดรูปเข้าตาราง Database ตรงๆ Database จะโหลดหนัก ช้า และเปลืองค่าเซิร์ฟเวอร์มหาศาลครับ การฝากไฟล์ที่ S3 ซึ่งออกแบบมาจัดการไฟล์โดยเฉพาะ จึงเป็น Best Practice แล้ว DB เราเก็บแค่ 'ลิ้งก์ URL ของรูปนั้น' ก็พอครับ"

## 📍 ส่วนที่ 2: เทคโนโลยีฝั่งหน้าบ้าน (Frontend Tech Stack)

**🗣️ สิ่งที่ต้องพูดพรีเซนต์:**
> "สำหรับฝั่ง Frontend หรือส่วนหน้าเว็บที่ผู้ใช้เข้าถึง เราเลือกใช้ไลบรารีชื่อดังอย่าง **React** ซึ่งเรานำมาครอบด้วย Framework ที่ชื่อว่า **Next.js** อีกชั้นหนึ่งครับ 
> 
> - **เหตุผลที่เลือก Next.js:** เพราะช่วยให้ระบบทำงานได้รวดเร็วมาก มีระบบจัดหน้า (Routing) ที่สะดวก และรองรับฝั่งเซิร์ฟเวอร์ได้ดีเยี่ยม
> - **สไตล์และการตกแต่งเว็บ (UI/UX):** เราเลือกใช้ **Tailwind CSS** ในการออกแบบหน้าตาให้สวยงามและรองรับมือถือ (Responsive)
> - **การเชื่อมต่อรับส่งข้อมูล:** เราใช้ **Axios** ในการยิง Request ไปพูดคุยกับหลังบ้านครับ"

**💡 คำถามดักทางที่อาจารย์มักจะถาม (Q&A):**
- **Q: ทำไมต้องใช้ Next.js ทำไมไม่ใช้ React เปล่าๆ เพียวๆ?**
  - **A ตอบอาจารย์:** "React เปล่าๆ จะเป็นแบบ SPA (Single Page Application) ซึ่งทำงานฝั่ง Client 100% แต่ Next.js มีฟีเจอร์ช่วยให้เราทำระบบต่างๆ ได้ดีกว่า เช่น `App Router` ที่จัดการโครงสร้าง URL ง่ายขึ้น รวมถึงมีฟีเจอร์ Edge Middleware ที่ช่วยเช็คสิทธิ์ (Auth Guard) ของผู้ใช้ก่อนที่จะอนุญาตให้เข้าหน้าหลักเลย ทำให้เว็บโหลดไวและปลอดภัยขึ้นมากครับ"
- **Q: Tailwind CSS คืออะไร ทำไมไม่เขียน CSS ปกติใส่คลาส?**
  - **A ตอบอาจารย์:** "Tailwind เป็น Utility-first CSS ครับ แทนที่เราจะต้องสลับพิมพ์ไฟล์ `.css` ไปมา เราสามารถใส่คำสั่งเช่น `flex p-4 bg-gray-100` ลงไปในแท็ก HTML หรือ Component ได้เลย ช่วยให้เขียนได้ไวมาก ควบคุมดีไซน์โปรเจคให้ไปในทิศทางเดียวกัน และเอาคลาสที่ไม่ได้ใช้ออกตอน Build อัตโนมัติ (เล็กและเบา) ครับ"

## 📍 ส่วนที่ 3: เทคโนโลยีฝั่งหลังบ้าน (Backend Tech Stack)

**🗣️ สิ่งที่ต้องพูดพรีเซนต์:**
> "มาที่ฝั่ง Backend หรือสมองของระบบที่เป็นตัวจัดการตรรกะทั้งหมด เราพัฒนาโปรเจคนี้ด้วยภาษา **TypeScript** ครับ โดยใช้เซิร์ฟเวอร์ **Node.js** ร่วมกับ Framework ที่ชื่อเรารู้จักกันดีว่า **NestJS** ครับ
> 
> - **ทำไมถึงเป็น NestJS:** เพราะ Framework นี้มีสถาปัตยกรรม (Architecture) ที่บังคับให้โค้ดเกิดระเบียบ จัดแบ่งเป็น "Module" (Controller/Service) อย่างชัดเจน ทำให้ระบบดูแลรักษาง่ายเวลาโครงสร้างใหญ่ขึ้น
> - **การยืนยันตัวตน (Authentication):** เราใช้ระบบ **JWT (JSON Web Token)**
> - **การอัปโหลดไฟล์:** เราใช้ **Multer** ในการเป็นตัวตรวจจับไฟล์รูป และใช้ **AWS SDK** เพื่ออัปโหลดไฟล์นั้นพุ่งตรงขึ้นไปเก็บบนระบบคลาวด์ครับ"

**💡 คำถามดักทางที่อาจารย์มักจะถาม (Q&A):**
- **Q: TypeScript มันต่างกับ JavaScript ตรงไหน ทำไมถึงเลือกใช้?**
  - **A ตอบอาจารย์:** "TypeScript เข้ามาแก้ปัญหาเจ็บปวดของ JavaScript ครับ คือมันช่วยเช็คชนิดของตัวแปร (Type Checking) ตั้งแต่ตอนเขียนโค้ด (Compile time) ถ้าโค้ดฝั่งเรารับค่าตัวเลข แต่วันนึงหน้าบ้านส่งข้อความ (String) มา TypeScript จะฟ้อง Error ให้แก้ทันทีก่อนขึ้นระบบจริง ช่วยกำจัดบั๊กของโปรแกรมไปได้เยอะมากครับเมื่อเทียบกับ JS ธรรมดา"
- **Q: JWT ทำงานยังไง อธิบายให้ฟังหน่อย?**
  - **A ตอบอาจารย์:** "เวลาผู้ใช้ Login กรอกรหัสถูกต้อง Backend จะเสก "ตั๋วผ่านทาง (Token)" หน้าตาเป็นตัวอักษรยาวๆ ส่งกลับไปให้หน้าบ้าน ตัวโทเค่นนี้จะมีการประทับตรา (Sign) ลายเซ็นลับไว้ 
เวลาหน้าบ้านจะทำอะไร (เช่น ขอข้อมูลตั๋ว) ก็แค่ชูโทเค่นนี้แนบมาด้วย Backend ก็จะตรวจลายเซ็นว่ายังไม่หมดอายุและเป็นของจริงไหม ถ้าจริงก็อนุญาต โดยไม่ต้องแวะไปเช็คฐานข้อมูลให้เสียเวลาทุกครั้งครับ"

## 📍 ส่วนที่ 4: ฐานข้อมูลการทำงาน (Database & ORM)

**🗣️ สิ่งที่ต้องพูดพรีเซนต์:**
> "สำหรับระบบจัดเก็บข้อมูล ข้อมูลผู้ใช้ และสถานะปัญหาทั้งหมด ระบบของเราใช้ฐานข้อมูลรูปแบบเชิงสัมพันธ์ หรือ **PostgreSQL** ครับ
> 
> โดยสิ่งที่เป็นตัวกลางเชื่อประหว่างโค้ด TypeScript ของเราให้คุยกับ PostgreSQL ได้ง่ายๆ ก็คือเครื่องมือที่เรียกว่า **Prisma ORM** ครับ Prisma จะรับหน้าที่แปลคำสั่งง่ายๆ ของเราให้กลายเป็นภาษา Database ที่ถูกต้อง รวมถึงป้องกันการโจมตีระบบแบบ SQL Injection ได้ด้วยครับ"

**💡 คำถามดักทางที่อาจารย์มักจะถาม (Q&A):**
- **Q: ORM คืออะไร ทำไมรันคำสั่ง Database (Raw SQL) สดๆ ไม่ดีกว่าเหรอ?**
  - **A ตอบอาจารย์:** "ORM (Object-Relational Mapping) มีข้อดีคือเราไม่ต้องจำโค้ดคำสั่ง SQL ซับซ้อนๆ ครับ เราแค่สั่งดึงข้อมูลด้วยภาษา TypeScript ปกติ Prisma ก็จะแปลให้ ความสุดยอดคือ Prisma มันต่อเข้ากับ TypeScript ได้มันเลยรู้ว่าตารางนั้นมีช่อง (Field) อะไรบ้าง ทำให้เกิด Auto-complete พิมพ์แล้วเด้งขึ้นมาเลยช่วยให้งานเสร็จไวและลด Human error ครับ"
- **Q: ทำไมเลือก PostgreSQL ไม่ใช้พวก NoSQL อย่าง MongoDB ละ?**
  - **A ตอบอาจารย์:** "ระบบ Helpdesk หรือตั๋วแจ้งซ่อม เป็นข้อมูลที่มีความสัมพันธ์กันชัดเจนม๊ากก (Clear Relational Data) เช่น "ตั๋ว 1 ใบ" เชื่อมโยงว่าเป็นของ "พนักงาน A" และถูกส่งต่อให้ "แผนก IT คนที่ B" ดังนั้นการใช้ Database รูปแบบตาราง (SQL) อย่าง PostgreSQL จึงตอบโจทย์เรื่องความถูกต้องของข้อมูล (Data integrity) ได้ดีกว่า การเก็บเป็น Document อิสระอย่าง MongoDB ครับ"
