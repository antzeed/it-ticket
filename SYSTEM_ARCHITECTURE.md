# 📘 สถาปัตยกรรมระบบ IT Helpdesk อย่างละเอียด (Comprehensive System Architecture)

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายการทำงานของระบบ IT Helpdesk รวบรวมตั้งแต่แนวคิด โครงสร้างโค้ด การเชื่อมต่อ ไปจนถึงการนำระบบขึ้นใช้งานจริงบนระบบคลาวด์ (AWS) โดยถูกออกแบบมาให้ **"ผู้เริ่มต้น หรือผู้ที่ไม่มีพื้นฐานโปรแกรมมิ่งเลย"** สามารถอ่านและทำความเข้าใจภาพรวมได้ทั้งหมดประหนึ่งว่ากำลังอธิบายโครงสร้างของบริษัทแห่งหนึ่ง

## 1. 🏗️ สถาปัตยกรรมภาพรวม (System Architecture) พี่ bent กับ A ต้องเข้าใจ

### 1.1 โครงสร้างฝั่งโค้ด (Code Design)
ระบบนี้ถูกออกแบบในลักษณะ **Client-Server Architecture** (ผู้ขอบริการ - ผู้ให้บริการ)
เปรียบเทียบง่ายๆ เหมือนการไปร้านอาหาร:
- **Client (หน้าบ้าน):** เหมือน "ลูกค้า" ที่ดูเมนูและสั่งอาหารผ่าน "พนักงานรับออเดอร์"
- **Server (หลังบ้าน):** เหมือน "พ่อครัว" ในห้องครัวที่รับออเดอร์ ไปหาวัตถุดิบ ปรุงอาหาร แล้วส่งกลับมาให้ลูกค้า

### 1.2 โครงสร้างฝั่งคลาวด์ (AWS Cloud Architecture)
เพื่อให้ระบบออนไลน์และรองรับคนจำนวนมากได้อย่างปลอดภัย เรานำระบบหลักไปวางบน **Amazon Web Services (AWS)** และเชื่อมออกสู่อินเทอร์เน็ตด้วย **Cloudflare Tunnel** แทนการเปิดพอร์ตตรงๆ:
1. **Cloudflare Tunnel (อุโมงค์ความปลอดภัย):** แทนที่จะพึ่งพาการเช่า ALB และ Route53 ของ AWS แบบเดิม เราใช้การทำงานผ่านท่อ Tunnel ของ Cloudflare เจาะช้องออกสู่อินเทอร์เน็ตเอง (Zero Trust Network) ช่วยให้โดเมนโหลดไว ซ่อน IP จริงของเซิร์ฟเวอร์ และป้องกันการโดนถล่มเว็บ (DDoS) 
2. **ECS Fargate (ห้องทำงานล่องหน):** ที่รันระบบ Frontend และ Backend ของเรา Fargate คือบริการที่ AWS เตรียมคอมพิวเตอร์ให้อัตโนมัติ เราแค่แพ็คโค้ดไปก้อนนึง มันรันให้เลยแบบ Serverless
3. **RDS PostgreSQL (ห้องนิรภัยเก็บข้อมูล):** เซิร์ฟเวอร์ฐานข้อมูลเฉพาะทางบน AWS ที่มีความปลอดภัยสูงและแบ็คอัพรายวันให้อัตโนมัติ
4. **S3 Bucket (โกดังเก็บไฟล์):** หน้าที่เหมือน Google Drive ไว้เก็บไฟล์แนบต่างๆ ที่ User อัปโหลดเข้ามา (เช่น รูปจอฟ้า, สลิป) เพราะการยัดรูปใส่ในตาราง Database ตรงๆ จะทำให้ระบบช้าและเปลืองค่าใช้จ่ายมาก
5. **ECR - Elastic Container Registry (โกดังเก็บแม่พิมพ์):** ที่สำหรับเก็บ "อิมเมจระบบ" (Docker Image) ของเรา เพื่อให้ ECS หยิบไปเปิดรัน
<img width="629" height="687" alt="image" src="https://github.com/user-attachments/assets/a7c4401c-3801-4f77-8176-0ac41401daf7" />
1. Cloudflare Tunnel (Zero Trust): คุณวางคอนเทนเนอร์ ECS Service: cloudflare (cloudflared daemon) ไว้ใน Cluster เดียวกัน ให้มันทำหน้าที่เจาะอุโมงค์ออกไปหาเซิร์ฟเวอร์ Cloudflare ข้างนอก ทำให้ไม่ต้องเปิดพอร์ต (Inbound) ขาเข้าเลย ปลอดภัยสุดๆ
2. การคุยกันภายใน (Call API): ECS Service: WEB สื่อสารหน้าบ้านและทำการส่ง Request/Proxy ไปหา ECS Service: API ได้โดยตรงผ่านเครือข่ายภายในของ VPC
3. ฐานข้อมูล (RDS): ECS Service: API เป็นตัวเดียวที่ต่อตรงเข้าไปหา PostgreSQL ปลอดภัยจากการถูกเข้าถึงจากภายนอก
4. การจัดการไฟล์ (S3): เห็นได้ชัดเจนว่ามีการเรียกใช้จากทั้ง API (ตอนอัปโหลดรูป) และ WEB (ตอนดึงรูปไปโชว์ในหน้าจอ)
5. การดึง Source (ECR): Cluster ทำการ Pull ท่อ Image มาจาก ECR
6. ระบบ Logging (CloudWatch): อันนี้ดีมากๆ ที่ใส่มาด้วยครับ เพราะ ECS Fargate เซิร์ฟเวอร์มันล่องหน การที่เราจะดู Log ว่าพังหรือไม่พัง ก็ต้องดูจากการพ่น Log ส่งไปรวมกันที่ CloudWatch ตรงนี้ทำให้ Flow ของ Diagram สมบูรณ์แบบสำหรับงาน Production เลยครับ

## 2. 🧩 ส่วนประกอบหลัก (Frontend, Backend, Database) ทำงานเชื่อมกันอย่างไร?

ระบบประกอบด้วย 3 ส่วนหลัก ซึ่งเราใช้ภาษา **TypeScript (JavaScript แบบมีระเบียบ)** ในการเขียนทั้งหมด (Full-stack TypeScript) ช่วยให้ทำงานง่ายขึ้นเพราะใช้ภาษาเดียวกันทั้งหน้าบ้านและหลังบ้าน

### 2.1 หน้าบ้าน (Frontend) - ใช้เทคโนโลยี Next.js (React) bow กับ pond ต้องเข้าใจ
- **หน้าที่:** เป็นหน้าตาของเว็บ (UI) สร้างปุ่ม สร้างฟอร์มให้กรอก แสดงผลสีสันต่างๆ รับคำสั่งจากผู้ใช้ (คลิก, พิมพ์)
- **การเปรียบเทียบ:** เหมือน **"พนักงานต้อนรับ"** ที่คอยพูดคุยกับลูกค้า รับเรื่อง แต่อนุมัติอะไรเองไม่ได้ ต้องวิ่งไปควักเอกสารหรือขอข้อมูลจากหลังบ้าน
- **ความพิเศษ:** เราใช้ **Next.js App Router** ร่วมกับ **TailwindCSS** ช่วยให้โหลดหน้าเว็บได้รวดเร็วมาก และเขียนสไตล์ (CSS) ได้ง่ายและสวยงาม

### 2.2 หลังบ้าน (Backend) - ใช้เทคโนโลยี NestJS (Node.js) พี่pondกับไอnack ต้องเข้าใจ
- **หน้าที่:** เป็นมันสมองของระบบ มีหน้าที่ตรวจสอบความถูกต้อง (เช่น รหัสผ่านถูกไหม? มีสิทธิ์ลบตั๋วใบนี้ไหม?) คอยรับเรื่องจากหน้าบ้าน ไปค้นข้อมูลในโกดัง (Database) และส่งผลลัพธ์กลับไป
- **การเปรียบเทียบ:** เหมือน **"ผู้จัดการ / ฝ่ายประมวลผล"** ที่ลูกค้ามองไม่เห็น ทำงานอยู่หลังร้าน มีหน้าที่ตัดสินใจและจัดการคลังสินค้าแบบเคร่งครัด
- **ความพิเศษ:** NestJS ถูกออกแบบมาให้โค้ดมีระเบียบ จัดแบ่งเป็น "ห้องๆ" (Modules) อย่างชัดเจน เช่น ห้อง Auth (จัดการล็อกอิน), ห้อง Ticket (จัดการตั๋ว)

### 2.3 ฐานข้อมูล (Database) - ใช้เทคโนโลยี PostgreSQL และ Prisma ORM ไอโต้ต้องเข้าใจ
- **หน้าที่:** จัดเก็บข้อมูลทุกอย่างที่ไม่ใช่ไฟล์ภาพ (เช่น ชื่อผู้ใช้, รหัสผ่านที่เข้ารหัสแล้ว, ข้อความในตั๋ว, สถานะตั๋ว)
- **Prisma ORM คืออะไร?** ปกติการคุยกับฐานข้อมูลต้องใช้ภาษา SQL ที่ซับซ้อน Prisma เปรียบเหมือน **"ล่ามแปลภาษา"** ที่ให้เราเขียนโค้ด TypeScript ธรรมดาๆ แล้วมันจะแปลเป็นคำสั่งไปดึงข้อมูลจากตารางให้เองโดยอัตโนมัติ

### 🔄 ตัวอย่างการทำงานร่วมกัน (Flow: แจ้งซ่อมคอมพิวเตอร์) ทุกคนควรเข้าใจ
1. **User (พนักงาน):** เปิดเข้าเว็บ (รันด้วย Next.js) พิมพ์รายละเอียดปัญหา กดแนบรูปจอฟ้า และกด "ส่ง Ticket"
2. **Frontend:** รับข้อมูลและรูปภาพ จัดการแพ็คใส่กล่อง แล้วส่งไปเคาะประตูเรียก **Backend** ผ่านทางเส้นทาง (API Endpoint) ที่ชื่อว่า `POST /api/tickets`
3. **Backend:** 
   - รับกล่องมา เปิดดูว่าคนส่งล็อกอินหรือยัง? (เช็คกุญแจ JWT)
   - เจอรูปภาพ 👉 Backend เอาตัวรูปภาพโยนขึ้นฟ้าไปเก็บที่ **AWS S3**
   - S3 เก็บรูปเสร็จ โยน URL (ลิ้งก์ของรูป) กลับมาให้ Backend
   - Backend เอา "ลิ้งก์รูป" + "รายละเอียดปัญหา" ไปบันทึกลงใน **Database (PostgreSQL)**
4. **Database:** บันทึกเสร็จ ตอบกลับว่า "เรียบร้อยจ้า บันทึกในเอกสารหมายเลข 1234"
5. **Backend:** หันไปบอก **Frontend** ว่า "สร้าง Ticket สำเร็จแล้ว หมายเลข 1234"
6. **Frontend:** โชว์หน้าต่างป๊อปอัพสีเขียวบอก User ว่า "แจ้งซ่อมสำเร็จ!" พร้อมพากลับไปหน้าแรก

## 3. 🐳 Docker และ Dockerfile คืออะไร? เอกับพี่bent

ปัญหาคลาสสิคของโปรแกรมเมอร์คือ **"เครื่องผมรันได้นะ แต่ทำไมไปรันเครื่องเซิร์ฟเวอร์แล้วพัง?"** (เพราะเวอร์ชั่นโปรแกรมในเครื่องไม่ตรงกัน)

**Docker** เปรียบเหมือน **"ตู้คอนเทนเนอร์สำเร็จรูป"** ที่เราจัดเตรียม สภาพแวดล้อม (OS), โปรแกรม (Node.js), และโค้ดของเรา แพ็คปิดผนึกรวมกันเป็นก้อนเดียว ยกก้อนนี้ไปเปิดที่ไหนในโลก ก็จะทำงานได้หน้าตาเหมือนกัน 100%

### 3.1 Dockerfile ของ Backend (API)
- **เหมือนใบสั่งประกอบเครื่องจักร:** ใบสั่งนี้บอกว่า 
  1. ให้โหลด Linux เปล่าๆ มา (Alpine) 
  2. ติดตั้ง Node.js 
  3. ก๊อปปี้ไฟล์โค้ดของเราใส่ลงไป 
  4. สั่งติดตั้ง dependencies (`npm install`) 
  5. รัน `prisma generate` (เตรียมล่ามแปลภาษา) 
  6. สั่งรันให้ระบบเริ่มทำงาน
- **ข้อดี:** AWS ECS จะทำตามใบสั่งนี้เป๊ะๆ เราจึงมั่นใจว่าหลังบ้านจะไม่พังเพราะขาดโปรแกรมอะไรไป

### 3.2 Dockerfile ของ Frontend (Web)
- โครงสร้างคล้ายหลังบ้าน แต่ตอนรันมันจะทำการ Build หน้าเว็บให้เป็นไฟล์ขนาดเล็กที่สุด (Optimization) เพื่อให้โหลดเร็วที่สุดตอนผู้ใช้เปิดเข้าเว็บ

## 4. 🔐 ตัวแปรสภาพแวดล้อม (.env / Environment Variables)

**ทำไมถึงต้องมี .env?**
ในโค้ดของเรา จะต้องมีการใช้รหัสผ่านฐานข้อมูล (Database Password) หรือกุญแจห้องนิรภัยของ AWS (AWS Secret Access Key)
การพิมพ์รหัสเหล่านั้น "ฝังลงไปในโค้ดตรงๆ" เป็นเรื่องที่ **อันตรายมาก** เพราะเมื่อเราเอาโค้ดขึ้น Git (ระบบเก็บประวัติโค้ด) ทุกคนในทีมหรือแฮกเกอร์จะเห็นรหัสผ่านทั้งหมด

**`.env` คือ ทางออก:**
มันคือไฟล์พิเศษ (เปรียบเหมือนกระดาษโพสต์อิทแผ่นเล็กๆ) ที่เราจดรหัสผ่านไว้แปะไว้ที่เครื่องเซิร์ฟเวอร์ โดยเราจะตั้งกฎ (`.gitignore`) ว่า **ห้ามเอาไฟล์ .env ขึ้น Git เด็ดขาด**

### ในระบบเรามีการแบ่ง .env อย่างไร? (เพื่อให้ปลอดภัยที่สุด)

1. **ฝั่งเว็บหน้าบ้าน (Root `.env`)** pond กับ โบว์ต้องเข้าใจ
   - มีแค่ตัวแปรเดียวคือ `NEXT_PUBLIC_API_URL` หรือ `API_URL`
   - **ทำไม?** เพราะหน้าเว็บคือสิ่งที่เบราว์เซอร์ของ User มองเห็น หากเราเอารหัสผ่านฐานข้อมูลมาใส่ฝั่งนี้ คนทั่วไปกดดู Source Code หน้าเว็บก็อาจจะโดนขโมยรหัสได้ ระบบจึงให้หน้าบ้านรู้จักแค่ว่า "เป้าหมายหลังบ้านอยู่ที่ลิ้งก์ไหน" เท่านั้น

2. **ฝั่งระบบหลังบ้าน (`apps/api/.env`)** พี่ปอนกับไอnack
   - เก็บความลับทั้งหมดของบริษัท!
   - `DATABASE_URL`: แผนที่และรหัสผ่านเพื่อไขเข้าห้องเก็บข้อมูล (PostgreSQL)
   - `JWT_SECRET`: รหัสลับสุดยอดที่หลังบ้านใช้ "ปั๊มตราประทับ" ลงในบัตรคิว (Token) เพื่อปล่อยให้ User เข้าสู่ระบบได้ (ถ้าแฮกเกอร์ได้กุญแจนี้ไป จะสามารถปลอมตัวเป็นใครก็ได้รวมถึงแอดมิน)
   - `AWS_ACCESS_KEY_ID` & `AWS_SECRET_ACCESS_KEY`: บัตรพนักงานของ AWS ที่ให้สิทธิ์หลังบ้านของเราเอาไฟล์รูปภาพโยนเข้าไปขว้างใส่ระบบ AWS S3 Bucket ได้

เวลาเราเอาของขึ้น Production (AWS ECS) เราก็แค่ไปกรอกค่าตัวแปร (Environment Variables) ตามโพสต์อิทฉบับนี้ ลงไปในหน้าตั้งค่าของ AWS โดยไม่ต้องแตะต้องหรือแก้โค้ดเลยแม้แต่บรรทัดเดียว เป็นอันเสร็จสิ้นพร้อมใช้งานจริงแบบมีความปลอดภัยระดับองค์กร!
