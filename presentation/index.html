Prisma Adapter & Connection
วิธีเชื่อมต่อแอปกับฐานข้อมูล — ใช้ Prisma ORM ผ่าน @prisma/adapter-pg

🔌 วิธีเชื่อมต่อฐานข้อมูล

TypeScript // 1. สร้าง connection pool (กลุ่มการเชื่อมต่อ DB) const pool = new Pool({ connectionString }); // 2. สร้าง adapter เพื่อให้ Prisma คุยกับ PostgreSQL const adapter = new PrismaPg(pool); // 3. ส่ง adapter ให้ PrismaClient ใช้งาน super({ adapter });
⚡ ทำไมต้อง adapter? ปกติ Prisma จะเชื่อม DB ตรงๆ แต่เราใช้ adapter ผ่าน pg connection pool เพื่อจัดการการเชื่อมต่อได้ดีขึ้น รองรับ serverless/edge environments

📦 โครงสร้าง Module ฝั่ง Backend

AppModule ← โมดูลหลักที่รวมทุกอย่าง
├── PrismaModule ← จัดการการเชื่อมต่อฐานข้อมูล
├── AuthModule ← ระบบ login/ยืนยันตัวตน (JWT)
├── TicketsModule ← สร้าง/ดู/แก้ไข ticket
└── UploadModule ← อัปโหลดรูปภาพไป AWS S3
💡 Module คืออะไร? NestJS แบ่งโค้ดเป็น Module แต่ละอันรับผิดชอบงานเฉพาะทาง ทำให้โค้ดเป็นระเบียบและดูแลง่าย

Auth Module — ระบบยืนยันตัวตน
ระบบยืนยันตัวตนด้วย JWT — ผู้ใช้ต้อง login ก่อนถึงจะใช้ระบบได้

🔐 ขั้นตอนการ Login

1
Browser ส่ง POST /auth/login พร้อม email + password
2
หา User จาก email ใน DB
3
เช็ค password (bcrypt หรือ plain-text)
4
สร้าง JWT token (payload: {sub, email, role})
5
ส่ง { accessToken } กลับ

🛡️ JWT Strategy — ตรวจสอบทุก request

1
ดึง JWT token จาก header Authorization: Bearer <token>
2
ตรวจสอบว่า token ถูกต้องด้วย secret key (กุญแจลับ)
3
ดึงข้อมูล user จาก DB โดยใช้ userId ที่ฝังอยู่ใน token (payload.sub)
4
ใส่ข้อมูล user ลง req.user เพื่อให้ controller อื่นๆ ใช้ต่อได้

API Endpoints — Auth & Tickets
เส้นทาง URL ที่ Frontend ใช้คุยกับ Backend — ทุก Tickets endpoint ต้อง login ก่อน

🔐 Auth — เส้นทางยืนยันตัวตน

MethodPathGuardคำอธิบาย
POST /auth/login ❌ ส่ง email+password → ได้ JWT token กลับ
GET /auth/me ✅ JWT ดึงข้อมูลผู้ใช้ที่ login อยู่ (ชื่อ, role, แผนก)

🎫 Tickets — เส้นทางจัดการ ticket

MethodPathสิทธิ์คำอธิบาย
POST /tickets ALL สร้าง ticket ใหม่ (ผูก authorId จาก JWT อัตโนมัติ)
GET /tickets ALL ADMIN: ทั้งหมด / USER: ของตัวเอง
GET /tickets/:id ALL ดูรายละเอียด ticket เฉพาะใบ
PATCH /tickets/:id ADMIN อัปเดต status, รับเรื่อง (assign)
💡 Endpoint คืออะไร? คือ URL ที่ Frontend ส่ง request ไปหา Backend เพื่อดึงหรือส่งข้อมูล เช่น GET /tickets = ขอดูรายการ ticket, POST /tickets = สร้าง ticket ใหม่

RBAC Logic & Upload Module
ระบบควบคุมสิทธิ์ตามบทบาท (Role-Based Access Control) + อัปโหลดรูปภาพไป AWS S3

🔒 RBAC Logic — ระบบควบคุมสิทธิ์

TypeScript // findAll — ดึงรายการ ticket ทั้งหมด (กรองตาม role) if (user.role === 'ADMIN') { return ทุก ticket // Admin เห็นทุกใบ } else { return ticket ที่ authorId === user.id // User เห็นแค่ของตัวเอง } // findOne — ดูรายละเอียด ticket (เช็คสิทธิ์ก่อน) if (user.role !== 'ADMIN' && ticket.authorId !== user.id) { throw ForbiddenException // ไม่ใช่เจ้าของ → ห้ามดู } // update — อัปเดต ticket ได้เฉพาะ ADMIN เท่านั้น if (user.role !== 'ADMIN') { throw ForbiddenException // User ทั่วไป → ห้ามแก้ไข }
💡 สรุป: ADMIN เห็นทุก ticket + แก้ไขได้ / USER เห็นแค่ ticket ตัวเอง + แก้ไขไม่ได้

📁 ขั้นตอนอัปโหลดรูปภาพ → AWS S3

1
Multer รับไฟล์ (เก็บใน memory — ไม่บันทึกลง disk)
2
ตรวจสอบ: เฉพาะ jpg/jpeg/png/gif ขนาด ≤ 5MB
3
S3Service.uploadFile() — สร้าง key: uploads/{uuid}.{ext}
4
PutObjectCommand — ส่งไฟล์ขึ้น AWS S3 Bucket
5
ส่ง URL กลับ: https://{bucket}.s3.{region}.amazonaws.com/...

⚙️ ตั้งค่า S3 (Environment Variables)

  • AWS_S3_REGION — ภูมิภาค (default: us-east-1)
  • AWS_S3_BUCKET_NAME — ชื่อ bucket
  • AWS_ACCESS_KEY_ID — คีย์เข้าถึง
  • AWS_SECRET_ACCESS_KEY — คีย์ลับ

API Proxy & Token Management
กลไกสำคัญ: Frontend ส่ง request ไป Backend อย่างไร + จัดการ token อย่างไร

🔀 API Proxy — ตัวกลางส่งต่อ request

1
Browser เรียก GET /api/tickets บน port 3000
2
Next.js middleware ตัด /api ออก → /tickets
3
สร้าง URL: http://api:4000/tickets
4
NextResponse.rewrite() → NestJS Backend
💡 ทำไมต้องทำแบบนี้?
Browser เรียก /api/* บน same origin (port เดียวกันคือ 3000) → ไม่มีปัญหา CORS (ข้อจำกัดของ browser ที่ห้ามเรียก URL ข้าม port)
Next.js ทำหน้าที่เป็นตัวกลาง ส่งต่อ request ไปยัง Backend โดย browser ไม่รู้ตัว

🍪 Token Management — จัดการตั๋วเข้าระบบ

1
Login สำเร็จ → เก็บ JWT token ไว้ใน Cookie ชื่อ "it_ticket_token" (หมดอายุ 1 วัน ต้อง login ใหม่)
2
ทุกครั้งที่เรียก API → ระบบดึง token จาก Cookie แล้วแนบไปกับ request อัตโนมัติ (ใส่ใน header Authorization)
3
ถ้า Backend ตอบ 401 (token หมดอายุ/ผิด) → ลบ token ทิ้ง → พาไปหน้า login ใหม่

Complete Flow — ตั้งแต่สร้างถึงจัดการ
Flow สมบูรณ์: User สร้าง Ticket → Admin จัดการ

👤 User สร้าง Ticket พร้อมรูปภาพ

1
Login ที่ /login → POST /api/auth/login → ได้ JWT → เก็บ Cookie
2
เข้า /tickets → UserDashboard → กด "สร้าง ticket ใหม่"
3
เลือกหมวดหมู่ + กรอกรายละเอียด + แนบรูป + เลือกความสำคัญ
4
Upload รูป: POST /api/upload → S3 → ได้ URL กลับ
5
สร้าง ticket: POST /api/tickets → Prisma INSERT → PostgreSQL
6
Redirect ไป HISTORY view → เห็น ticket ที่สร้าง

👨‍💼 Admin จัดการ Ticket

1
Admin login → เข้า /tickets → เห็น AdminDashboard
2
เห็น ticket ทุกใบ (role=ADMIN → backend ส่งทั้งหมด)
3
กรอง status + department ← client-side filtering
4
กด "Manage" → ไปหน้า /tickets/[id]
5
"Assign to me": PATCH → UPDATE assigneeId
6
เปลี่ยน Status: PATCH → UPDATE status (OPEN → IN_PROGRESS → CLOSED)
